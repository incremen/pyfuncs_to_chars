pyfuncs_to_chars

represent any character using only python builtin function calls

The constraint: write a Python expression that evaluates to any character, using only builtin function calls. No numeric literals, no string literals, no operators. Each function takes exactly one argument.

So chr(97) is out (uses a number). ord('a') is out (uses a string). But ord(max(str(bytes()))) is fine — it evaluates to 98 using only function calls on function calls.

The question is: can you reach every Unicode character this way? And if so, how short can the expression be?

We need two operations. Together they let us build any number.

Subtract 1: max(range(n)) returns n-1.

range(n) produces 0, 1, ..., n-1. max() picks the last one. Costs 2 parentheses.

Multiply by 3: len(str(list(bytes(n)))) returns exactly 3n.

bytes(n) creates n zero-bytes. list() turns it into [0, 0, ..., 0]. str() gives "[0, 0, ..., 0]" — always exactly 3n characters. Costs 4 parentheses.

The algorithm: to build n, recursively build ceil(n/3), triple it, then subtract 0, 1, or 2. This is base-3 decomposition. The recursion stops at a base anchor — a small number constructible directly from builtins (like len(str(not())) = 4). 

# example: build 13
# 13 in base 3 is 111 (1*9 + 1*3 + 1)

build(13):
  ceil(13/3) = 5,  remainder = 3*5 - 13 = 2
  13 = 3 * build(5) - 2

build(5):
  ceil(5/3) = 2,  remainder = 3*2 - 5 = 1
  5 = 3 * build(2) - 1

build(2):
  base anchor: len(str(ord(min(str(not())))))

# assembling: 13 = triple(triple(anchor) - 1) - 2

Python has a 200 nested parentheses limit. The base-3 approach uses at most ~114 for the largest Unicode code point (1,114,111), so it fits.

The base-3 formula works for everything, but isn't always the shortest. Other strategies can produce shorter expressions for specific numbers. The optimizer tries all of them and keeps the best.

Different ways to multiply, each with a different paren cost:

len(str(list(bytes(n))))= 3n4 parens
len(str(bytes(n)))= 4n + 33 parens
len(ascii(str(bytes(n))))= 5n + 54 parens

4n+3 costs only 3 parens (cheaper than 3x at 4), so it's often better when you can land on the right value with the +3 offset.

ascii() escapes backslashes. Stacking k calls on str(bytes(n)) gives:

f(n) = (2^k + 3) * n + (2^(k+1) + 1)

Exponential multiplier for linear paren cost.

zip() wrapping creates nested tuples, growing the string repr by 3n each time:

len(str(list(zip^k(bytes(n))))) = 3(k+1) * n

k=16n5 parens
k=29n6 parens
k=312n7 parens
k=415n8 parens

Clean exact multipliers with no offset. Useful when you need an exact multiple of 6, 9, 12, etc.

Triangular: sum(range(n)) = n(n-1)/2. Quadratic growth, 2 parens.

Log step-down: len(str(n)) = digit count. Collapses a big number to something small. 2 parens.

Boolean collapse: int(bool(n)) = 1 for any n>0. Reset to 1. 2 parens.
